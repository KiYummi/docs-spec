# 残差审计指南

> **Version:** V 1.0
> **Last Updated:** 2026-05-16
> **适用范围:** 文档工程体系（docs-spec）

---

## 一、审计目标

残差审计的目标是确保全链路信号保真度，即：

1. **需求完整性**：每条原子需求在下游文档中都有承接
2. **术语一致性**：字段名和术语符合字典定义
3. **数据流一致性**：模块依赖遵循架构设计
4. **短路校验**：原始需求直通底层制品，验证完整性

---

## 二、核心原理

### 2.1 跳跃连接（Skip Connection）

**概念**：借鉴残差网络的跳跃连接思想，下游文档必须直接锚定上游原文，避免信号逐层衰减。

**实现**：
- 下游文档（LLD/API 合约/数据模型）必须包含"需求溯源区"
- 溯源区直接引用上游原文（PRD/Sub-PRD），不经过中间层
- 溯源区内容由人工或工具填充，LLM 不得修改

**效果**：
- 需求信号不会在文档层级间衰减
- 每个设计决策都可追溯到原始需求
- 审计时可以快速定位偏差来源

### 2.2 残差信号（Residual Signal）

**概念**：借鉴残差网络的残差学习思想，审计时关注"偏差"而非"绝对值"。

**实现**：
- 语义覆盖率审计：检查需求要素是否在下游有承接
- 术语一致性扫描：检查字段名和术语是否符合字典
- 数据流一致性检查：检查模块依赖是否遵循架构

**效果**：
- 偏差可检测：覆盖率、一致率、违例数
- 偏差可追溯：每个偏差都有明确来源
- 偏差可修正：偏差可以转化为结构化补丁

### 2.3 补丁式修正（Patch Correction）

**概念**：借鉴残差网络的恒等映射思想，修改文档时只修改偏差部分，不重写全文。

**实现**：
- 修改文档必须以结构化补丁形式提交
- 禁止直接重写文档正文
- 补丁必须包含：reason、source_requirement、impact

**效果**：
- 修改可追溯：每个修改都有原因和来源
- 修改可回滚：补丁可以反向应用
- 修改可审计：补丁可以被审计

## 三、审计步骤

### 3.1 提取基线

1. 从 PRD 中提取原子需求清单
2. 每条需求分配唯一 ID（@req-id: R001）
3. 语义要素必须完整：谁、什么、何时、何地、为什么

### 3.2 逐层比对

1. **PRD → Sub-PRD**：检查每条需求是否在 Sub-PRD 中有承接
2. **Sub-PRD → LLD**：检查每条需求是否在 LLD 中有承接
3. **LLD → API**：检查每条需求是否在 API 合约中有承接
4. **API → 数据库**：检查每条需求是否在数据模型中有承接
5. **数据库 → 测试**：检查每条需求是否在测试用例中有承接

### 3.3 短路校验

1. 原始需求直通底层制品，验证完整性
2. 检查：PRD → Sub-PRD → LLD → API → 数据库 → 测试
3. 每一层必须有对应的需求 ID 标注

### 3.4 输出补丁

1. 将审计发现转化为结构化补丁
2. 补丁格式遵循 `docs/_schemas/patch_schema.json`
3. 补丁必须包含：reason、source_requirement、impact

## 四、推理链要求（CoT）

### 4.1 推理链格式

```
发现：{问题描述}
证据：{文件路径}:{行号} → {具体内容}
推理：{推理过程}
结论：{明确结论}
```

### 4.2 推理链示例

```
发现：需求 R001 在 LLD 中未找到对应承接
证据：docs/00_meta/master-prd.md:15 → "用户可以通过手机号注册"
推理：PRD 中定义了 R001 需求，但在 LLD 中搜索 "手机号" 和 "注册" 相关内容，未找到对应的设计文档
结论：LLD 缺少 R001 需求的设计，需要补充
```

### 4.3 推理链规则

- 每个审计结论必须包含推理过程
- 不接受"经验判断"式的结论
- 推理链必须可追溯

## 六、增量审计（卷积·局部感受野）

### 6.1 概念

借鉴卷积网络的局部感受野思想，增量审计只检查变更的部分及其下游，不全量扫描。

### 6.2 增量审计规则

1. **识别变更范围**：Git diff 或 Coordinator 指定
2. **分析下游影响**：变更的文档 → 下游文档 → 下下游文档
3. **只检查变更部分及其下游**，不全量扫描
4. **输出增量审计报告**

### 6.3 增量审计流程

```
1. 识别变更范围（Git diff 或 Coordinator 指定）
   ↓
2. 分析下游影响（变更的文档 → 下游文档 → 下下游文档）
   ↓
3. 只检查变更部分及其下游
   ↓
4. 输出增量审计报告
   ↓
5. 记录审计日志
```

### 6.4 增量审计优势

- **效率**：只检查变更部分，大幅减少审计时间
- **精度**：聚焦变更影响，减少噪音
- **可操作**：审计结果更聚焦，更容易转化为补丁
