## 二、纵向判据 — 层间边界（DATA / {平台层} / {应用层}）

### 2.1 核心原则：谁存、谁加工、谁决策

| 层 | 一句话定位 | 写什么 | 不写什么 |
|---|-----------|--------|---------|
| **DATA** | 存 — 被多模块共享的结构化数据 | 表结构、字段定义、校验规则、RBAC、种子数据、消费关系 | 业务流转、状态机（简单字典状态除外）、渲染逻辑、UI 描述、领域事件 |
| **{平台层}** | 加工 — 横向技术能力 | 引擎架构、API 契约、配置数据结构、NFR、数据依赖、被消费关系 | 业务实体的表定义（→ DATA）、业务决策逻辑（→ {应用层}）、具体业务流程编排 |
| **{应用层}** | 决策 — 端到端业务流程 | 状态机、业务规则、审批校验计算、业务单据表、UI 交互、数据依赖、引擎依赖 | 模板渲染/文件解析/LLM 推理（→ {平台层}）、共享基础数据表定义（→ DATA）、引擎级编排逻辑（→ {平台层}） |

### 2.2 依赖方向

```

{应用层} → DATA → shared

反向禁止。每层只写本层内容，跨层依赖用指向，不重复定义。
```

## 五、使用规则

### 5.1 强制遵循

- 所有 Agent **必须**遵循本规范的分层边界
- 禁止跨层重复定义
- 发现边界冲突必须先注册到本规范，再处理

### 5.2 更新流程

1. 发现边界冲突
2. 提交到规范审核
3. 审核通过后注册到规范
4. 通知所有 Agent

### 5.3 冲突处理

- 同一概念使用相同归属
- 发现冲突以本规范为准
- 冲突由 Coordinator 决策
