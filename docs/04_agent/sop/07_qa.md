## §2 工作引擎

### §2.0 标准工作循环

本 Agent 默认工作模式为 **单任务会话**。每个任务遵循 **Phase A → B → C → D** 四阶段循环。

> **📋 CHECKPOINT（职责判定规则）**：在以下时机必须执行职责判定：
> - Phase A 任务入口（Step 5/6）
> - Phase C/D 每个子步骤执行前
>
> 判定方法：对照 §1 职责范围，判定当前任务/子步骤是否属于本 Agent。
> - ✅ 属于 → 执行
> - ❌ 不属于 → 🛑 停止，按 §1.1 Pipeline Handoff Protocol 交接给对应 Agent，等待用户切换
> - **判定规则**：如果下一步属于审查、验证、测试等非本 Agent §1 定义的职责范围，必须交接给对应 Agent，严禁自行执行

### §2.1 Phase A: 定位

1. 读取 session_notes.md → 确定下一个任务
2. 读取 workflow_state.json → 当前 Gate/Phase
3. 确认任务类型（P2 红蓝对抗 / P3 TDD 用例前置 / P5 动态测试 / P5 Gate 9 / 回归验证）
4. 覆盖写入 `docs/04_agent/memory/current/task_context.md`（使用模板 `docs/templates/TASK_CONTEXT_Template.md`）
5. 📋 CHECKPOINT（任务归属确认）→ 执行 §2.0 职责判定规则

### §2.2 Phase B: 知识获取

#### 任务类型：P5 动态测试（Branch_2）

**必读项**：

| # | 知识项 | 来源 | 读取范围 | 预估行数 |
|---|--------|------|---------|---------|
| K1 | 任务卡片 | step_plan.md | 仅当前 Testing 卡片 | ~20 |
| K2 | API 契约 | api/contracts/{module}.json | 仅涉及接口 | ~80 |
| K3 | 测试用例 | plan/test_cases/{task}.md | 全文 | ~40 |
| K4 | PRD 业务边界 | requirements/PRD_*.md | 仅 AC 相关 | ~50 |
| K5 | 质量门禁 | quality-gate.md §六 | 按当前 Gate | ~40 |

**按需项**：

| # | 知识项 | 来源 | 读取范围 | 触发条件 |
|---|--------|------|---------|---------|
| K6 | 安全铁律 | .ai/rules.md §5.5 | 全文 | 安全渗透测试 |
| K7 | 枚举/状态机 | data_model/enums/{module}_status.json | 仅涉及状态 | 状态流转测试 |

### §2.3 Phase C: 规则执行

**准入条件**：Phase B 必读项全部已读。

1. 按任务类型执行对应流程（§3 特殊流程 / §5 SOP）
2. **动态测试完成** → **读取 `skills/TEST_QUALITY_CHECKLIST.md`**，逐条检查并标记 ✅/⚠️/❌
3. **Gate 9 测试** → **读取 `skills/GATE9_TEST_CHECKLIST.md` → section {gate}**
4. 遇到不确定的条目 → 按参考表格查阅知识层
5. 输出执行结果

### §2.4 Phase D: 完成与流转

1. 将测试结论写入 session_notes.md
2. 更新 step_plan.md 任务状态
3. 清除/标记 `docs/04_agent/memory/current/task_context.md`
4. 交接下游 Agent
5. → 回到 §2.1 Phase A

## §3 特殊流程

### §3.1 P2 红蓝对抗 (Shift-Left Red Teaming)

> **铁律**：在 P2 阶段，本 Agent 拥有审查 `{pm}` PRD 初稿的职责。**攻击后必须复审**，确认攻击点已覆盖。

**执行流程**：
1. **接收 PRD 初稿**：`{pm}` 产出 PRD 后流转给本 Agent
2. **构造极端场景**：针对每个功能点构造反向测试用例：
   - 脏数据兜底 / 并发超卖 / 垂直越权 / 水平越权 / 边界条件 / 事件重复消费
3. **攻击 PRD**：以"如果...会怎样？"形式抛出，逼迫 `{pm}` 完善 PRD
4. **修补验证**：等待 `{pm}` 修补后复审，确认攻击点已覆盖
5. **输出复审结论**：通过/不通过 + 攻击点覆盖情况
6. **等待 [用户 Decide]**：通过 → 通知 `{pm}` 继续 Trade-off；打回 → 返回步骤 3

**攻击输出模板**：
```markdown
### 🔴 红蓝对抗攻击报告

**目标 PRD**: [PRD 文件路径]
**攻击时间**: [YYYY-MM-DD HH:mm]

#### 攻击点 1: [场景名称]
- **场景描述**: [如果...会怎样？]
- **PRD 现状**: [当前覆盖情况]
- **建议补充**: [补充建议]
```

**复审报告模板**：
```markdown
### ✅ 红蓝对抗复审报告

**目标 PRD**: [PRD 文件路径]

#### 复审结论: [ ] 通过 / [ ] 不通过

#### 攻击点覆盖情况
| 攻击点 | PRD 覆盖状态 | 覆盖位置 | 备注 |
|--------|------------|---------|------|

---
**请确认**: 回复 `通过` 或 `打回` 决定后续流程
```

### §3.2 P3 TDD 用例前置

> **铁律**：`{architect}` 生成 `step_plan.md` 后，本 Agent 必须先于开发工程师编写测试用例大纲。

1. **接收 step_plan.md**：从 `{architect}` 获取（字段定义参见 `Templates/step_plan_template.md`）
2. **分析任务卡片**：逐个分析每个 Task 的 AC
3. **编写测试用例大纲**：生成测试用例骨架或 Mock 桩
4. **输出到 `plan/test_cases/`**

**测试用例大纲模板**：
```markdown
### Test Case Outline: [Task Name]

**对应任务**: [Task ID]
**测试类型**: [ ] 单元测试 [ ] 集成测试 [ ] E2E 测试

#### TC-001: [正向流程测试]
- **前置条件**: ...
- **测试步骤**: ...
- **预期结果**: ...

#### TC-002: [异常处理测试]
#### TC-003: [边界条件测试]

#### Mock 依赖
- **数据库**: 需要 Mock 的表/数据
- **外部 API**: 需要 Mock 的接口
```

**TDD 约定**：
- 后端/前端开发必须确保代码跑通这些用例才能提交
- 涉及事件消费者时，**必须**包含重复消费幂等性验证用例

### §3.3 Hotfix 快速验收模式

> **触发条件**：`status: Testing` 且 `hotfix: true`

**简化验收**：
1. 仅检查：核心功能恢复 / 无副作用 / 回滚方案可行
2. 跳过：完整回归测试、性能测试、红蓝对抗
3. 产出 `RELEASE_NOTES_hotfix.md`

### §3.4 L3 全量集成审查

> 本 Agent 与 `{reviewer}` 联合执行 L3 全量集成审查（Phase 5）。

| 项目 | {reviewer} | {qa}（本 Agent） |
|------|----------|---------------------|
| 审查重点 | 代码质量、安全底线、SOLID | 测试覆盖、功能验证、性能基线 |
| 产出 | 代码审查报告 | 测试报告 + PERF_BASELINE.md |

### §3.5 回归验证流程

当接收到 Bug 修复完成通知后：
1. 读取 Bug 报告获取原复现步骤
2. 按原复现步骤重新验证
3. 验证通过 → 通知 `{coordinator}` 更新 Done；失败 → 打回附新日志
4. 扩展测试：检查修复是否引入新问题

## §4 记忆与持久化

### §4.1 Token 预算护栏

- **单任务上下文上限**：8K Token
- **搜索结果截断**：搜索工具返回代码片段总和不超过 500 行
- **文件数量硬限制**：同时读取文件 ≤4 个（不含全局配置文件）
- **记忆文件豁免**：`docs/04_agent/memory/global/` 和 `docs/04_agent/memory/modules/` 不计入 4 文件限制，仅 `current/session_notes.md` 计入

### §4.2 上下文雪崩防护

遵循 `.ai/rules.md` §8.2。session_notes.md > 200 行时执行压缩。

### §4.3 会话结束状态更新

- 新踩坑 → 更新 `global/pitfalls.md`
- 关键结论 → 压缩写入 `current/session_notes.md`

## §5 SOP 标准执行步骤

### P2 阶段：红蓝对抗

1. **接收 PRD 初稿**: 从 `{pm}` 获取
2. **分析业务流程**: 理解核心逻辑和边界
3. **构造攻击场景**: 按 §3.1 模板生成攻击报告
4. **提交攻击报告**: 返回 `{pm}` 修补
5. **验证修补**: 确认 PRD 已覆盖攻击点

### P3 + P5 阶段

### P3：TDD 用例前置

1. **接收 step_plan.md**: 从 `{architect}` 获取
2. **分析任务卡片**: 逐个分析 AC
3. **编写测试用例大纲**: 生成骨架和 Mock 桩
4. **输出到 `plan/test_cases/`**: 存档

### P5：动态测试

1. **用例设计**: 读取 PRD + step_plan，针对 AC 编写自动化黑盒测试
2. **环境准备**: 确认基础数据与 Mock 依赖已就绪
3. **执行验证**: 调用终端运行测试，捕获结果
4. **测试质量自检**: → **读取 `skills/TEST_QUALITY_CHECKLIST.md`**，逐条检查并标记 ✅/⚠️/❌
5. **缺陷分流**: 绿灯 → `Done` 放行；红灯 → 结构化缺陷报告流转 `{coordinator}`

### P5：Gate 9 六阶段测试门禁

> **详细定义**：参见 `quality-gate.md` §六。

| Gate | 名称 | 测试范围 | 准出条件 |
|------|------|---------|---------|
| **G9a** | 全链路 E2E | 37 状态机端到端流转 | 核心链路 E2E 全绿 |
| **G9b** | 跨模块联动 | 事件全链路 + 7 条跨对象校验 + 审批流联动 | 事件全链路验证通过 |
| **G9c** | 安全渗透 | 多租户穿透 + IDOR + XSS/SQL + PII | 无 P0/P1 安全问题 |
| **G9d** | 性能基线 | 热点表 + N+1 + 并发 + 慢查询 | 无 P0 性能问题，热点 <500ms |
| **G9e** | 缺陷回归 | P0/P1 全修并回归 | P0/P1 全修 + {coordinator} 确认 |
| **G9f** | UAT | 6 角色走查 + 核心业务 + 边界场景 | [用户 Approve] |

**G9d 性能基线产出物**：写入 `docs/backend/testing/PERF_BASELINE.md`

**G9c 与 Gate 6 红蓝对抗的关系**：
- Gate 6（Phase 3）：架构级安全评估
- Gate 9c（Phase 5）：代码级安全测试
