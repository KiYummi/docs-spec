## 4. 原子需求提取 [强制]

> **说明**：PRD 完成后，必须提取原子需求清单。每条原子需求分配唯一 ID（@req-id: R001）。
> 原子需求是全链路索引的起点，所有下游制品必须标注承接的需求 ID。
> 详见 `docs/08_templates/traceability-zone.md`。

### 4.1 原子需求清单

| ID | 需求描述 | 语义要素 | 来源 | 优先级 |
|----|---------|---------|------|--------|
| @req-id: R001 | {需求描述} | {要素1, 要素2, ...} | {来源章节} | P0/P1/P2 |
| @req-id: R002 | {需求描述} | {要素1, 要素2, ...} | {来源章节} | P0/P1/P2 |

### 4.2 提取规则

- 一条原子需求只描述一个功能点
- 语义要素必须完整：谁、什么、何时、何地、为什么
- 来源必须标注到 PRD 的具体章节
- 优先级必须与 PRD 一致
- **Grounding**：原子需求必须逐字复制 PRD 原文，不得改写

### 4.3 语义要素说明

| 要素 | 说明 | 示例 |
|------|------|------|
| 谁 | 需求的主体（用户/系统/角色） | 管理员、普通用户、系统 |
| 什么 | 需求的动作（操作/功能） | 创建、查询、删除、审批 |
| 何时 | 需求的时间约束 | 实时、定时、T+1 |
| 何地 | 需求的场景/模块 | 用户管理、订单管理、报表 |
| 为什么 | 需求的目的/价值 | 提高效率、降低风险、满足合规 |
