# 需求溯源区模板

> **用途**：下游文档（LLD/API 合约/数据模型）必须包含此模板，实现跳跃连接
> **填充方式**：由人工或工具填充，LLM 不得修改
> **来源**：PRD/Sub-PRD 中的原子需求清单

---

## 溯源区格式

```markdown
## 需求溯源区

> **说明**：本章节为需求溯源区，直接锚定上游原文（PRD/Sub-PRD），实现跳跃连接。
> 溯源区内容由人工或工具填充，LLM 不得修改。

### 原子需求承接

| 原子需求 ID | 语义要素 | 来源（原文） | 推导链 |
|------------|---------|-------------|--------|
| @req-id: R001 | {要素描述} | {PRD/Sub-PRD 原文} | {设计决策推理过程} |
| @req-id: R002 | {要素描述} | {PRD/Sub-PRD 原文} | {设计决策推理过程} |

### 推导链模板

#### 推导链 1：@req-id: R001

**需求**：{PRD 原文}

**约束**：
- {约束 1}
- {约束 2}

**方案**：
- {方案描述}

**理由**：
- {为什么选择这个方案}
- {与其他方案的对比}

**结论**：
- {最终设计决策}
```

## 使用说明

### 1. 何时使用

- LLD（低层设计文档）
- API 合约
- 数据模型文档
- 其他下游技术文档

### 2. 如何填充

1. 从 PRD/Sub-PRD 中提取原子需求清单
2. 将每条需求的 ID、语义要素、原文填入表格
3. 为每个设计决策编写推导链（CoT）
4. 标注术语约束
5. 汇总索引标注

### 3. 填充规则

- **Grounding**：溯源区内容必须逐字复制上游原文，不得改写
- **完整性**：溯源区必须覆盖所有相关原子需求
- **推导链**：每个设计决策必须包含推理过程（CoT）
- **术语一致**：字段名必须符合 metadata-dictionary，术语必须符合 glossary

### 4. 审计检查

残差审计时会检查以下内容：

- [ ] 溯源区是否存在
- [ ] 溯源区是否覆盖所有相关原子需求
- [ ] 溯源区内容是否逐字复制上游原文（Grounding）
- [ ] 每条需求是否标注了唯一 ID（@req-id: R001）
- [ ] 每个设计决策是否包含推理链（CoT）
- [ ] 字段名是否符合 metadata-dictionary
- [ ] 术语是否符合 glossary
