# Sub-PRD → LLD 流程

> **适用范围**：文档工程体系（docs-spec）
> **最后更新**：2026-05-16
> **涉及 Agent**：docs, architect, coordinator, residual-auditor
> **涉及思想**：跳跃连接、Grounding、CoT

---

## 一、流程概述

```
Sub-PRD（子需求文档）
    ↓
docs/architect：填写溯源区
    ↓
docs/architect：编写推导链
    ↓
人：确认溯源区
    ↓
docs/architect：生成 LLD
    ↓
residual-auditor：审计
    ↓
人：确认审计结果
```

---

## 二、详细步骤

### 步骤 1：填写溯源区（docs/architect）

**职责**：docs/architect 在 LLD 中填写需求溯源区

**操作**：
1. 读取 Sub-PRD 中的原子需求清单
2. 在 LLD 中创建"需求溯源区"章节
3. 将每条需求的 ID、语义要素、原文填入表格
4. **Grounding**：溯源区内容必须逐字复制上游原文，不得改写

**输出**：

```markdown
## 需求溯源区

| 原子需求 ID | 语义要素 | 来源（原文） | 推导链 |
|------------|---------|-------------|--------|
| @req-id: R001 | {要素描述} | {Sub-PRD 原文} | {设计决策推理过程} |
```

**规则**：
- 溯源区必须覆盖所有相关原子需求
- 溯源区内容必须逐字复制上游原文
- 溯源区由人工或工具填充，LLM 不得修改

### 步骤 2：编写推导链（docs/architect）

**职责**：docs/architect 为每个设计决策编写推导链

**操作**：
1. 识别每个设计决策
2. 编写推导链：需求 → 约束 → 方案 → 理由
3. 推导链必须可追溯

**输出**：

```markdown
### 推导链 1：@req-id: R001

**需求**：{Sub-PRD 原文}

**约束**：
- {约束 1}
- {约束 2}

**方案**：
- {方案描述}

**理由**：
- {为什么选择这个方案}
- {与其他方案的对比}

**结论**：
- {最终设计决策}
```

**规则**：
- 每个设计决策必须包含推理链
- 推理链必须可追溯
- 不接受"经验判断"式的结论

### 步骤 3：人确认溯源区（人在回路）

**职责**：人确认溯源区的完整性和准确性

**操作**：
1. 检查溯源区是否覆盖所有相关原子需求
2. 检查溯源区内容是否逐字复制上游原文
3. 检查推导链是否完整
4. 确认或修改

**规则**：
- 人有最终决策权
- 人可以修改、删除、新增内容
- 人确认后才可继续

### 步骤 4：生成 LLD（docs/architect）

**职责**：docs/architect 根据溯源区生成 LLD

**操作**：
1. 读取溯源区内容
2. 设计数据模型（ER/枚举/状态机）
3. 设计 API 接口
4. 设计业务逻辑
5. 生成 LLD 文档

**输出**：LLD 文档

**规则**：
- LLD 必须承接溯源区中的所有需求
- 数据模型必须符合 metadata-dictionary
- API 接口必须符合 API 契约规范

### 步骤 5：审计（residual-auditor）

**职责**：residual-auditor 审计 LLD 的保真度

**操作**：
1. 提取原子需求基线
2. 检查语义覆盖率
3. 检查术语一致性
4. 检查数据流一致性
5. 输出审计报告

**输出**：审计报告 + 补丁建议

**规则**：
- 审计必须覆盖所有原子需求
- 审计结论必须包含推理链
- 补丁必须遵循 patch_schema.json

### 步骤 6：人确认审计结果（人在回路）

**职责**：人确认审计结果和补丁建议

**操作**：
1. 检查审计报告
2. 审批补丁建议
3. 确认或修改

**规则**：
- 人有最终决策权
- 人可以审批、拒绝、修改补丁
- 人确认后才可继续

## 三、跳跃连接实现

### 3.1 概念

借鉴残差网络的跳跃连接思想，LLD 必须直接锚定 Sub-PRD 原文，避免信号逐层衰减。

### 3.2 实现方式

1. **溯源区**：LLD 中包含"需求溯源区"，直接引用 Sub-PRD 原文
2. **推导链**：每个设计决策包含推理过程，可追溯到原始需求
3. **Grounding**：溯源区内容必须逐字复制上游原文，不得改写

### 3.3 效果

- 需求信号不会在文档层级间衰减
- 每个设计决策都可追溯到原始需求
- 审计时可以快速定位偏差来源
