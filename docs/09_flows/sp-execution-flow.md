# SP 执行流程（MOE 团队模型）

> **适用范围**：文档工程体系（docs-spec）
> **最后更新**：2026-05-16
> **涉及 Agent**：coordinator, pm, architect, docs, backend, frontend, reviewer, residual-auditor
> **涉及思想**：MOE（混合专家模型）、门控机制、稀疏激活

---

## 一、流程概述

```
coordinator：接收任务
    ↓
coordinator：判断任务类型（门控）
    ↓
coordinator：选择执行 Agent（稀疏激活）
    ↓
执行 Agent：执行任务
    ↓
reviewer：审查结果
    ↓
residual-auditor：审计保真度
    ↓
coordinator：汇总结果
    ↓
人：确认
```

## 二、MOE 团队模型

### 2.1 角色映射

| 原角色 | MOE 角色 | Agent | 职责 |
|--------|---------|-------|------|
| 调度+决策+记录 | 门控网络 | Coordinator | 根据 Phase 类型动态选择执行 agent；审查结果；决策问题；记录 |
| 执行（PRD 类） | 专家 1 | pm | 写 PRD 相关的 Phase 文档 |
| 执行（架构类） | 专家 2 | architect | 写架构相关的 Phase 文档 |
| 执行（技术文档类） | 专家 3 | docs | 写技术文档相关的 Phase 文档 |
| 执行（代码类） | 专家 4 | backend/frontend | 写代码相关的 Phase |
| 审查 | 专家 5 | reviewer | 检查 Phase 文档内容完整性、模板合规 |
| 审计 | 专家 6 | residual-auditor | 残差保真度审计 |

### 2.2 门控逻辑

**任务类型 → 执行 Agent**：

| 任务类型 | 调用的 Agent | 说明 |
|---------|-------------|------|
| PRD 类 | pm | 写 PRD 相关的文档 |
| 架构类 | architect | 写架构相关的文档 |
| 技术文档类 | docs | 写技术文档 |
| 代码类 | backend/frontend | 写代码 |
| 审查 | reviewer | 检查文档/代码质量 |
| 审计 | residual-auditor | 残差保真度审计 |

### 2.3 稀疏激活

- 每次只激活相关的 Agent，不是全部
- 不相关的 Agent 不参与当前任务
- 减少资源消耗，提高效率

## 三、详细步骤

### 步骤 1：接收任务（coordinator）

**职责**：coordinator 接收任务

**操作**：
1. 接收用户任务
2. 分析任务类型
3. 确定执行 Agent

### 步骤 2：判断任务类型（coordinator）

**职责**：coordinator 判断任务类型（门控）

**操作**：
1. 分析任务内容
2. 判断任务类型（PRD/架构/技术文档/代码/审查/审计）
3. 选择执行 Agent

### 步骤 3：选择执行 Agent（coordinator）

**职责**：coordinator 选择执行 Agent（稀疏激活）

**操作**：
1. 根据任务类型选择 Agent
2. 调用 `task(subagent_type="xxx")`
3. 传递任务上下文

### 步骤 4：执行任务（执行 Agent）

**职责**：执行 Agent 执行任务

**操作**：
1. 读取任务上下文
2. 执行任务
3. 输出结果

**输出**：任务结果

### 步骤 5：审查结果（reviewer）

**职责**：reviewer 审查结果

**操作**：
1. 检查任务结果
2. 检查内容完整性
3. 检查模板合规
4. 输出审查报告

**输出**：审查报告

### 步骤 6：审计保真度（residual-auditor）

**职责**：residual-auditor 审计保真度

**操作**：
1. 提取原子需求基线
2. 检查语义覆盖率
3. 检查术语一致性
4. 输出审计报告

**输出**：审计报告

### 步骤 7：汇总结果（coordinator）

**职责**：coordinator 汇总结果

**操作**：
1. 汇总执行结果
2. 汇总审查报告
3. 汇总审计报告
4. 输出汇总报告

**输出**：汇总报告

### 步骤 8：人确认（人在回路）

**职责**：人确认结果

**操作**：
1. 检查汇总报告
2. 确认或修改

**规则**：
- 人有最终决策权
- 人可以修改、删除、新增内容
- 人确认后才可继续

|-------|-------|------|

|-------|-------|------|

## 五、日志记录

### 5.1 决策日志

**触发时机**：每次人做决策时

**记录者**：coordinator

**日志格式**：

```markdown
# 决策日志

> **决策时间**：YYYY-MM-DD HH:MM:SS
> **决策背景**：{背景描述}
> **决策选项**：
> - 选项 A：{描述}
> - 选项 B：{描述}
> **决策**：{最终决策}
> **决策理由**：{理由}
> **影响范围**：{影响范围}
```

### 5.2 审计日志

**触发时机**：每次审计完成后

**记录者**：residual-auditor

**日志格式**：

```markdown
# 审计日志

> **审计时间**：YYYY-MM-DD HH:MM:SS
> **审计范围**：{模块/文档/需求}
> **审计员**：residual-auditor

## 审计发现

### 发现 1：{问题描述}
- **发现**：{问题描述}
- **证据**：{文件路径}:{行号} → {具体内容}
- **推理**：{推理过程}
- **结论**：{明确结论}
- **补丁建议**：{结构化补丁}

## 审计结论
- 覆盖率：{覆盖率}%
- 术语一致率：{一致率}%
- 数据流违例：{违例数}

## 反思
- 审计过程中的问题和改进建议
```

### 5.3 变更日志

**触发时机**：每次规则变更时

**记录者**：coordinator

**日志格式**：

```markdown
# 变更日志

> **变更时间**：YYYY-MM-DD HH:MM:SS
> **变更内容**：{变更描述}
> **变更前**：{变更前内容}
> **变更后**：{变更后内容}
> **预期效果**：{预期效果}
```
