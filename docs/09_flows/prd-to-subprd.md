# PRD → Sub-PRD 流程

> **适用范围**：文档工程体系（docs-spec）
> **最后更新**：2026-05-16
> **涉及 Agent**：pm, coordinator, residual-auditor
> **涉及思想**：RAG（生成前检索）、索引、Grounding

---

## 一、流程概述

```
PRD（主需求文档）
    ↓
pm：提取原子需求
    ↓
pm：分配唯一 ID
    ↓
人：确认原子需求
    ↓
pm：生成 Sub-PRD
    ↓
residual-auditor：审计
    ↓
人：确认审计结果
```

---

## 二、详细步骤

### 步骤 1：提取原子需求（pm）

**职责**：pm 从 PRD 中提取原子需求清单

**操作**：
1. 读取 PRD 全文
2. 按功能点拆分需求
3. 为每条需求分配唯一 ID（@req-id: R001）
4. 提取语义要素：谁、什么、何时、何地、为什么
5. 标注来源章节
6. 标注优先级（P0/P1/P2）

**输出**：

```markdown
## 原子需求清单

| ID | 需求描述 | 语义要素 | 来源 | 优先级 |
|----|---------|---------|------|--------|
| @req-id: R001 | {需求描述} | {要素1, 要素2, ...} | {来源章节} | P0/P1/P2 |
```

**规则**：
- 一条原子需求只描述一个功能点
- 语义要素必须完整
- 来源必须标注到 PRD 的具体章节
- 优先级必须与 PRD 一致
- **Grounding**：原子需求必须逐字复制 PRD 原文，不得改写

### 步骤 2：人确认原子需求（人在回路）

**职责**：人确认原子需求清单的完整性和准确性

**操作**：
1. 检查原子需求是否覆盖 PRD 所有功能点
2. 检查语义要素是否完整
3. 检查优先级是否正确
4. 确认或修改

**规则**：
- 人有最终决策权
- 人可以修改、删除、新增需求
- 人确认后才可继续

### 步骤 3：生成 Sub-PRD（pm）

**职责**：pm 根据原子需求生成 Sub-PRD

**操作**：
1. 读取原子需求清单
2. 按模块/功能分组
3. 生成 Sub-PRD 文档
4. 包含：功能描述、用户故事、验收标准、边界条件、非功能需求

**输出**：Sub-PRD 文档

**规则**：
- Sub-PRD 必须承接所有原子需求
- 每个功能点必须有对应的验收标准
- 边界条件和非功能需求必须覆盖

### 步骤 4：审计（residual-auditor）

**职责**：residual-auditor 审计 Sub-PRD 的保真度

**操作**：
1. 提取原子需求基线
2. 检查语义覆盖率
3. 检查术语一致性
4. 输出审计报告

**输出**：审计报告 + 补丁建议

**规则**：
- 审计必须覆盖所有原子需求
- 审计结论必须包含推理链
- 补丁必须遵循 patch_schema.json

### 步骤 5：人确认审计结果（人在回路）

**职责**：人确认审计结果和补丁建议

**操作**：
1. 检查审计报告
2. 审批补丁建议
3. 确认或修改

**规则**：
- 人有最终决策权
- 人可以审批、拒绝、修改补丁
- 人确认后才可继续

## 三、RAG 集成（生成前检索）

### 3.1 概念

借鉴 RAG（Retrieval-Augmented Generation）思想，在生成 Sub-PRD 前，先检索相关上下文。

### 3.2 检索内容

| 检索源 | 内容 | 用途 |
|--------|------|------|
| PRD | 原子需求清单 | 需求基线 |
| glossary | 术语定义 | 术语一致性 |
| metadata-dictionary | 字段定义 | 字段名一致性 |
| global-blueprint | 模块依赖 | 数据流一致性 |
| 历史 Sub-PRD | 相似需求 | 参考和复用 |

### 3.3 检索流程

```
1. 识别当前需求的关键词
   ↓
2. 检索相关文档（PRD/glossary/metadata-dictionary/global-blueprint）
   ↓
3. 提取相关内容
   ↓
4. 作为上下文注入生成过程
   ↓
5. 生成 Sub-PRD
```
