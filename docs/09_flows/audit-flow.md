# 残差审计流程

> **适用范围**：文档工程体系（docs-spec）
> **最后更新**：2026-05-16
> **涉及 Agent**：coordinator, residual-auditor
> **涉及思想**：残差网络、跳跃连接、补丁式修正、CoT、日志系统

---

## 一、流程概述

```
coordinator：触发审计
    ↓
residual-auditor：执行审计
    ↓
residual-auditor：输出审计报告
    ↓
residual-auditor：记录审计日志
    ↓
coordinator：审核结果
    ↓
人：签发补丁
    ↓
coordinator：应用补丁
```

---

## 二、详细步骤

### 步骤 1：触发审计（coordinator）

**职责**：coordinator 触发残差审计

**触发场景**：

| 场景 | 触发时机 | 说明 |
|------|---------|------|
| 文档生成后 | coordinator | 检查新生成文档的保真度 |
| Gate 通过前 | coordinator | 强制门禁检查 |
| 重大修订后 | coordinator | 检查修订是否引入偏差 |
| 定期巡检 | coordinator | 全链路健康检查 |

**操作**：
1. 确定审计范围（模块/文档/需求）
2. 调用 `task(subagent_type="residual-auditor")`
3. 传递审计指令

**输出**：审计指令

### 步骤 2：执行审计（residual-auditor）

**职责**：residual-auditor 执行残差审计

**操作**：
1. 读取审计指令
2. 提取原子需求基线
3. 执行审计（按维度逐项检查）：
   - 语义覆盖率审计
   - 术语一致性扫描
   - 数据流一致性检查
   - 短路校验
4. 输出审计报告
5. 输出补丁建议

**输出**：审计报告 + 补丁建议

**规则**：
- 审计必须覆盖所有原子需求
- 审计结论必须包含推理链
- 补丁必须遵循 patch_schema.json

### 步骤 3：记录审计日志（residual-auditor）

**职责**：residual-auditor 记录审计日志

**操作**：
1. 生成审计日志
2. 保存到 `docs/04_agent/logs/audits/`

**输出**：审计日志

**日志格式**：

```markdown
# 审计日志

> **审计时间**：YYYY-MM-DD HH:MM:SS
> **审计范围**：{模块/文档/需求}
> **审计员**：residual-auditor

## 审计发现

### 发现 1：{问题描述}
- **发现**：{问题描述}
- **证据**：{文件路径}:{行号} → {具体内容}
- **推理**：{推理过程}
- **结论**：{明确结论}
- **补丁建议**：{结构化补丁}

## 审计结论
- 覆盖率：{覆盖率}%
- 术语一致率：{一致率}%
- 数据流违例：{违例数}

## 反思
- 审计过程中的问题和改进建议
```

### 步骤 4：审核结果（coordinator）

**职责**：coordinator 审核审计结果

**操作**：
1. 检查审计报告
2. 检查补丁建议
3. 决定是否提交给人签发

**规则**：
- coordinator 有审核权
- coordinator 可以要求重新审计
- coordinator 可以修改补丁建议

### 步骤 5：签发补丁（人在回路）

**职责**：人签发补丁

**操作**：
1. 检查审计报告
2. 检查补丁建议
3. 审批、拒绝、修改补丁

**规则**：
- 人有最终决策权
- 人可以审批、拒绝、修改补丁
- 人签发后才可应用

### 步骤 6：应用补丁（coordinator）

**职责**：coordinator 应用补丁

**操作**：
1. 读取签发的补丁
2. 应用补丁到目标文件
3. 记录变更日志

**输出**：变更日志

**日志格式**：

```markdown
# 变更日志

> **变更时间**：YYYY-MM-DD HH:MM:SS
> **变更内容**：{变更描述}
> **变更前**：{变更前内容}
> **变更后**：{变更后内容}
> **预期效果**：{预期效果}
```

## 三、增量审计流程

### 3.1 概念

借鉴卷积网络的局部感受野思想，增量审计只检查变更的部分及其下游，不全量扫描。

### 3.2 流程

```
1. 识别变更范围（Git diff 或 Coordinator 指定）
   ↓
2. 分析下游影响（变更的文档 → 下游文档 → 下下游文档）
   ↓
3. 只检查变更部分及其下游
   ↓
4. 输出增量审计报告
   ↓
5. 记录审计日志
```

### 3.3 优势

- **效率**：只检查变更部分，大幅减少审计时间
- **精度**：聚焦变更影响，减少噪音
- **可操作**：审计结果更聚焦，更容易转化为补丁

## 四、Reflexion 集成（后审计反思）

### 4.1 概念

借鉴 Reflexion 思想，在审计完成后进行反思，改进审计规则。

### 4.2 反思流程

```
1. 审计完成
   ↓
2. 分析审计结果
   ↓
3. 识别审计过程中的问题
   ↓
4. 提出改进建议
   ↓
5. 更新审计规则
```

### 4.3 反思内容

- 审计覆盖率是否足够？
- 审计规则是否准确？
- 审计效率是否可优化？
- 审计结果是否可操作？
