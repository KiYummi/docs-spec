# docs-spec：可复用文档工程体系

> **版本**：v1.0
> **适用场景**：大型项目（>50 条需求、≥3 层文档传递、依赖 LLM 辅助生成）
> **核心目标**：在"人决策 + LLM 执行"的协作模式下，确保需求在多跳传递中不失控
> **理论基础**：工程控制论 + Karpathy + DDD + 残差网络 + 卷积网络 + 混合专家模型 + LLM 前沿思想

---

## 一、核心问题

在 LLM+人协作开发中，需求从 PRD 到最终代码落地，经历多跳传递（PRD → Sub-PRD → LLD → API → 代码），每一跳都可能引入偏差：

| 偏差类型 | 表现 | 根因 |
|---------|------|------|
| **语义衰减** | 原始需求的约束条件被遗漏 | LLM 概括时丢失细节 |
| **术语漂移** | 同一概念在不同文档中用不同名称 | 缺乏统一术语锚定 |
| **隐性偏移** | LLM "合理重述"导致微妙语义变化 | 人难以察觉自然语言的微小变化 |
| **连锁反应** | 一处修改未同步到关联文档 | 缺乏全链路追溯机制 |

## 二、理论基础

本体系融合六大理论支柱 + 五个 LLM 前沿思想：

| 理论 | 核心概念 | 在本体系中的应用 |
|------|---------|----------------|
| **工程控制论** | 反馈控制、稳定性、可观测性 | 闭环反馈、不振荡、状态可追溯 |
| **Karpathy Software 2.0** | 文档=代码、LLM=编译器 | 文档是规约源代码 |
| **DDD** | 限界上下文、通用语言、防腐层 | 模块边界、术语表、依赖隔离 |
| **残差网络** | 跳跃连接、残差信号、补丁修正 | 溯源区、偏差检测、增量修改 |
| **卷积网络** | 局部感受野、权重共享 | 增量审计、规则复用 |
| **MOE** | 专家分工、门控机制、稀疏激活 | Agent 专业化、动态调度 |
| **CoT** | 推理链 | 设计决策可追溯 |
| **RAG** | 检索增强生成 | 生成前检索上下文 |
| **Reflexion** | 后审计反思 | 审计后改进规则 |
| **Grounding** | 逐字复制 | 溯源区原文锚定 |
| **Active Learning** | 主动确认 | 不确定时主动提问 |

## 三、核心机制

### 3.1 跳跃连接（需求溯源区）

下游文档直接锚定上游原文，绕过中间层的概括与改写。

详见：[01_strategies/skip-connection.md](01_strategies/skip-connection.md)

### 3.2 全链路索引（原子需求 ID）

以 `@req-id: R001` 为全局键，建立双向追溯关系。

详见：[01_strategies/full-index.md](01_strategies/full-index.md)

### 3.3 补丁式修正（增量修改）

任何修改以结构化补丁形式提交，禁止直接重写文档。

详见：[01_strategies/patch-correction.md](01_strategies/patch-correction.md)

### 3.4 残差审计（信号保真度检查）

四维审计：语义覆盖率、术语一致性、数据流一致性、短路校验。

详见：[01_strategies/residual-audit.md](01_strategies/residual-audit.md)

### 3.5 增量审计（卷积思想）

只检查变更的部分及其下游，不全量扫描。

详见：[01_strategies/incremental-audit.md](01_strategies/incremental-audit.md)

## 四、Agent 体系（MOE 团队模型）

| Agent | 类型 | 职责 |
|-------|------|------|
| **coordinator** | primary | 门控调度、决策签发、日志记录 |
| **architect** | primary | 架构设计、LLD 编写 |
| **pm** | subagent | PRD 编写、原子需求提取 |
| **docs** | subagent | 技术文档编写、溯源区维护 |
| **backend** | subagent | 后端编码 |
| **frontend** | subagent | 前端编码 |
| **reviewer** | subagent | 代码审查、术语一致性检查 |
| **qa** | subagent | 测试执行、需求覆盖率验证 |
| **devops** | subagent | 部署运维 |
| **residual-auditor** | subagent | 残差保真度审计 |

详见：[agents/](agents/)

## 五、人在回路

| 层级 | 人的职责 | LLM 不能替代的原因 |
|------|---------|-------------------|
| **源头定义** | 确定原子需求清单 | 无法替你做取舍与价值判断 |
| **边界裁决** | 设计模板、定义术语边界 | 无法判断约束过严或过松 |
| **信号确认** | 审核溯源区是否被篡改 | 无法察觉自己"合理重述"导致的语义漂移 |
| **补丁签发** | 审核并批准/驳回补丁 | 无法预估跨领域的连锁影响 |
| **规则演化** | 根据高频偏差调整规则 | 无法判断何时需要修改体系本身 |

## 七、适用条件

| 条件 | 必要性 | 说明 |
|------|--------|------|
| 需求规模 >50 条 | 必要 | 小规模项目人脑可覆盖 |
| 多角色文档传递 ≥3 层 | 必要 | 单层传递无需跳跃连接 |
| 依赖 LLM 辅助生成 | 必要 | 人工编写偏差可控 |

### 裁剪策略

| 项目类型 | 建议组合 |
|---------|---------|
| **小型项目**（<20 需求） | 仅用"术语表 + 人工审查" |
| **中型项目**（20-50 需求） | 术语表 + 跳跃连接（关键需求） |
| **大型项目**（>50 需求） | 全量使用四机制 + 人在回路 |

## 八、快速开始

1. **克隆仓库**：`git clone https://github.com/KiYummi/docs-spec.git`
2. **阅读方法论**：本文件 + `01_strategies/` 目录
3. **选择模板**：从 `02_templates/doc-skeleton/` 选择需要的骨架
4. **定义术语**：创建项目的 `glossary.md` 和 `metadata-dictionary.md`
5. **提取需求**：使用 `04_tools/baseline-extractor.py` 提取原子需求
6. **开始迭代**：按 `docs/09_flows/` 中的流程执行
