# 残差审计检查清单

残差审计是文档工程体系的核心机制，确保全链路信号保真度。审计员执行时逐条检查。

## 审计维度

### 1. 语义覆盖率审计（残差信号）

**目标**：检查每条原子需求的语义要素是否在下游文档中有承接。

**检查项**：

- [ ] 原子需求基线 — PRD 中是否提取了原子需求清单？
- [ ] 语义要素覆盖 — 每条需求的语义要素是否在下游文档中有承接？
- [ ] 覆盖率计算 — 覆盖率 = 有承接的语义要素数 / 总语义要素数 × 100%
- [ ] 未覆盖标注 — 未覆盖的需求是否标注了原因？

**覆盖率要求**：
- P0 需求：100% 覆盖
- P1 需求：≥ 90% 覆盖
- P2 需求：≥ 70% 覆盖

### 2. 术语一致性扫描（残差信号）

**目标**：检查字段名和术语是否符合字典定义。

**检查项**：

- [ ] 字段名合规 — 字段名是否在 metadata-dictionary 中？
- [ ] 术语合规 — 术语是否在 glossary 中？
- [ ] 命名一致性 — 同一概念是否使用相同术语？
- [ ] 格式一致性 — 命名格式是否符合规范（camelCase/snake_case）？

### 3. 数据流一致性检查（残差信号）

**目标**：检查模块依赖是否遵循架构设计。

**检查项**：

- [ ] 模块依赖方向 — 是否遵循 global-blueprint？
- [ ] 数据流向 — 是否与架构设计一致？
- [ ] 循环依赖 — 是否存在循环依赖？
- [ ] 跨域调用 — 跨业务域是否使用事件模型（非同步调用）？

### 4. 短路校验（跳跃连接）

**目标**：原始需求直通底层制品，验证完整性。

**检查项**：

- [ ] PRD → Sub-PRD — 每条需求是否在 Sub-PRD 中有承接？
- [ ] Sub-PRD → LLD — 每条需求是否在 LLD 中有承接？
- [ ] LLD → API — 每条需求是否在 API 合约中有承接？
- [ ] API → 数据库 — 每条需求是否在数据模型中有承接？
- [ ] 数据库 → 测试 — 每条需求是否在测试用例中有承接？

### 5. 增量审计（卷积·局部感受野）

**目标**：只检查变更的部分及其下游，不全量扫描。

**检查项**：

- [ ] 变更范围识别 — 是否识别了变更范围（Git diff 或 Coordinator 指定）？
- [ ] 下游影响分析 — 是否分析了变更对下游的影响？
- [ ] 增量检查 — 是否只检查了变更部分及其下游？
- [ ] 增量报告 — 是否输出了增量审计报告？

**增量审计规则**：
1. 识别变更范围：Git diff 或 Coordinator 指定
2. 分析下游影响：变更的文档 → 下游文档 → 下下游文档
3. 只检查变更部分及其下游，不全量扫描
4. 输出增量审计报告

### 6. 推理链输出（CoT）

**目标**：每个审计结论必须包含推理过程。

**检查项**：

- [ ] 发现描述 — 是否清晰描述了发现？
- [ ] 证据引用 — 是否引用了具体证据（文件路径、行号）？
- [ ] 推理过程 — 是否包含了推理过程？
- [ ] 结论明确 — 是否给出了明确结论？

**推理链格式**：
```
发现：{问题描述}
证据：{文件路径}:{行号} → {具体内容}
推理：{推理过程}
结论：{明确结论}
```

### 7. 补丁建议格式（补丁式修正）

**目标**：将审计发现转化为结构化补丁。

**检查项**：

- [ ] 补丁格式 — 是否遵循 `docs/_schemas/patch_schema.json`？
- [ ] 原因说明 — 是否包含修改原因（reason）？
- [ ] 来源需求 — 是否标注了来源需求 ID（source_requirement）？
- [ ] 影响范围 — 是否分析了影响范围（impact）？
- [ ] 时间戳 — 是否包含时间戳（timestamp）？

## 审计流程

```
1. 接收审计指令（Coordinator）
   ↓
2. 识别审计范围（模块/文档/需求）
   ↓
3. 提取原子需求基线
   ↓
4. 执行审计（按维度逐项检查）
   ↓
5. 输出审计报告 + 补丁建议
   ↓
6. 记录审计日志
   ↓
7. 返回报告给 Coordinator
```

## 审计报告格式

```markdown
# 残差审计报告

## 审计范围
- 模块：{模块名}
- 文档：{文档路径}
- 需求：{需求 ID 范围}

## 审计发现

### 发现 1：{问题描述}
- **发现**：{问题描述}
- **证据**：{文件路径}:{行号} → {具体内容}
- **推理**：{推理过程}
- **结论**：{明确结论}
- **补丁建议**：{结构化补丁}

### 发现 2：{问题描述}
...

## 审计结论
- 覆盖率：{覆盖率}%
- 术语一致率：{一致率}%
- 数据流违例：{违例数}

## 补丁清单
| 补丁 ID | 目标文件 | 操作 | 原因 |
|---------|---------|------|------|
| P001 | {文件路径} | {replace/add/remove} | {原因} |
```

## 红旗

> 停止打勾，先解决：

- 语义覆盖率 < 70% → 先补全再继续
- 术语一致率 < 80% → 先统一术语再继续
- 数据流存在循环依赖 → 先重构再继续
- 推理链缺失 → 审计结论不可信，先补充推理链再继续
