# 补丁式修正策略详解

> **适用范围**：文档工程体系（docs-spec）
> **最后更新**：2026-05-16
> **涉及思想**：残差网络、补丁式修正

---

## 一、概念

### 1.1 残差网络中的恒等映射

在残差网络中，恒等映射（Identity Mapping）是一种特殊的映射方式，它将输入直接传递到输出，形成 `y = x` 的结构。

**作用**：
- 保留原始信号
- 只修改偏差部分
- 避免重写整个网络

### 1.2 文档工程体系中的补丁式修正

在文档工程体系中，补丁式修正的思想被应用到文档修改中：

**传统方式**：
- 直接重写文档正文
- 修改不可追溯
- 修改不可回滚

**补丁式修正方式**：
- 以结构化补丁形式提交
- 修改可追溯
- 修改可回滚

## 二、补丁格式

### 2.1 补丁结构

```json
{
  "id": "P001",
  "target": "docs/00_meta/master-prd.md",
  "op": "replace",
  "path": "## 1. 需求描述",
  "value": "新的内容",
  "oldValue": "旧的内容",
  "reason": "需求变更",
  "source_requirement": "@req-id: R001",
  "impact": ["docs/02_backend/specs/GLOBAL_HLD.md"],
  "timestamp": "2026-05-16T10:00:00Z",
  "author": "residual-auditor",
  "status": "pending"
}
```

### 2.2 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | 补丁 ID，格式：P{序号} |
| target | string | 目标文件路径 |
| op | string | 操作类型：replace/add/remove |
| path | string | JSON 指针或 Markdown 章节路径 |
| value | any | 新值 |
| oldValue | any | 旧值 |
| reason | string | 修改原因 |
| source_requirement | string | 来源需求 ID |
| impact | array | 影响范围 |
| timestamp | string | 时间戳 |
| author | string | 作者 |
| status | string | 状态：pending/approved/rejected/applied |

## 三、补丁操作类型

### 3.1 replace（替换）

**操作**：替换指定内容

**示例**：
```json
{
  "op": "replace",
  "path": "## 1. 需求描述",
  "value": "新的内容",
  "oldValue": "旧的内容"
}
```

### 3.2 add（新增）

**操作**：新增指定内容

**示例**：
```json
{
  "op": "add",
  "path": "## 2. 新增章节",
  "value": "新增内容"
}
```

### 3.3 remove（删除）

**操作**：删除指定内容

**示例**：
```json
{
  "op": "remove",
  "path": "## 3. 删除章节",
  "oldValue": "被删除的内容"
}
```

## 四、补丁流程

### 4.1 补丁生成

**触发时机**：残差审计发现偏差时

**操作**：
1. 审计发现偏差
2. 生成补丁建议
3. 输出补丁

**输出**：补丁建议

### 4.2 补丁审核

**触发时机**：coordinator 审核补丁时

**操作**：
1. 检查补丁格式
2. 检查补丁内容
3. 决定是否提交给人签发

**输出**：审核结果

### 4.3 补丁签发

**触发时机**：人签发补丁时

**操作**：
1. 检查补丁建议
2. 审批、拒绝、修改补丁
3. 签发补丁

**输出**：签发的补丁

### 4.4 补丁应用

**触发时机**：coordinator 应用补丁时

**操作**：
1. 读取签发的补丁
2. 应用补丁到目标文件
3. 记录变更日志

**输出**：变更日志

### 4.5 补丁回滚

**触发时机**：需要回滚补丁时

**操作**：
1. 读取补丁
2. 生成反向补丁
3. 应用反向补丁
4. 记录变更日志

**输出**：变更日志

## 五、效果

### 5.1 修改可追溯

- 每个修改都有原因和来源
- 每个修改都有时间戳和作者
- 每个修改都有状态

### 5.2 修改可回滚

- 补丁可以反向应用
- 可以回滚到任意版本
- 可以恢复到修改前状态

### 5.3 修改可审计

- 补丁可以被审计
- 补丁可以被签发
- 补丁可以被应用
