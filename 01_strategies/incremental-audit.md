# 增量审计策略详解（卷积思想）

> **适用范围**：文档工程体系（docs-spec）
> **最后更新**：2026-05-16
> **涉及思想**：卷积网络、局部感受野、增量审计

---

## 一、概念

### 1.1 卷积网络中的局部感受野

在卷积神经网络中，局部感受野（Local Receptive Field）是一种特殊的连接方式，每个神经元只连接输入的一个局部区域。

**作用**：
- 减少参数数量
- 提高计算效率
- 保留局部特征

### 1.2 文档工程体系中的增量审计

在文档工程体系中，增量审计的思想被应用到文档质量检查中：

**传统方式**：
- 全量扫描所有文档
- 检查所有需求
- 检查所有制品

**增量审计方式**：
- 只检查变更的部分
- 只检查变更的下游
- 只输出增量报告

**作用**：
- 提高审计效率
- 减少审计噪音
- 聚焦变更影响

## 二、增量审计规则

### 2.1 识别变更范围

**方式**：
- Git diff：通过 Git 差异识别变更
- Coordinator 指定：由 Coordinator 指定变更范围

**内容**：
- 变更的文件
- 变更的章节
- 变更的需求

### 2.2 分析下游影响

**方式**：
- 变更的文档 → 下游文档
- 下游文档 → 下下游文档
- 递归分析

**内容**：
- 受影响的文档
- 受影响的需求
- 受影响的制品

### 2.3 只检查变更部分及其下游

**方式**：
- 只检查变更的文档
- 只检查变更的下游
- 不检查未变更的部分

**内容**：
- 语义覆盖率
- 术语一致性
- 数据流一致性

### 2.4 输出增量审计报告

**方式**：
- 只输出变更部分的审计结果
- 只输出变更部分的补丁建议
- 不输出未变更部分的结果

**内容**：
- 变更部分的审计结果
- 变更部分的补丁建议
- 变更部分的影响范围

---

## 三、增量审计流程

### 3.1 识别变更范围

```
1. 识别变更范围（Git diff 或 Coordinator 指定）
   ↓
2. 提取变更的文件列表
   ↓
3. 提取变更的章节列表
   ↓
4. 提取变更的需求列表
```

### 3.2 分析下游影响

```
1. 变更的文档 → 下游文档
   ↓
2. 下游文档 → 下下游文档
   ↓
3. 递归分析
   ↓
4. 输出受影响的文档列表
```

### 3.3 只检查变更部分及其下游

```
1. 只检查变更的文档
   ↓
2. 只检查变更的下游
   ↓
3. 不检查未变更的部分
   ↓
4. 输出审计结果
```

### 3.4 输出增量审计报告

```
1. 只输出变更部分的审计结果
   ↓
2. 只输出变更部分的补丁建议
   ↓
3. 不输出未变更部分的结果
   ↓
4. 输出增量审计报告
```

## 四、增量审计示例

### 4.1 场景

**变更**：PRD 中的需求 R001 描述变更

**影响**：
- PRD → Sub-PRD
- Sub-PRD → LLD
- LLD → API
- API → 数据库
- 数据库 → 测试

### 4.2 增量审计

**步骤**：
1. 识别变更范围：PRD 中的 R001
2. 分析下游影响：Sub-PRD、LLD、API、数据库、测试
3. 只检查变更部分及其下游
4. 输出增量审计报告

**输出**：
- R001 在 Sub-PRD 中的承接情况
- R001 在 LLD 中的承接情况
- R001 在 API 中的承接情况
- R001 在数据库中的承接情况
- R001 在测试中的承接情况

## 五、效果

### 5.1 效率提升

- 只检查变更部分，大幅减少审计时间
- 不检查未变更部分，减少计算量
- 聚焦变更影响，减少噪音

### 5.2 精度提升

- 聚焦变更影响，减少误报
- 聚焦变更影响，减少漏报
- 聚焦变更影响，提高准确性

### 5.3 可操作性提升

- 审计结果更聚焦
- 补丁建议更精准
- 更容易转化为行动

## 六、与全量审计的对比

| 维度 | 全量审计 | 增量审计 |
|------|---------|---------|
| 范围 | 所有文档 | 变更的文档及其下游 |
| 时间 | 长 | 短 |
| 精度 | 高 | 高 |
| 噪音 | 高 | 低 |
| 可操作性 | 低 | 高 |
