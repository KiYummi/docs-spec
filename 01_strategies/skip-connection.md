# 跳跃连接策略详解

> **适用范围**：文档工程体系（docs-spec）
> **最后更新**：2026-05-16
> **涉及思想**：残差网络、跳跃连接

---

## 一、概念

### 1.1 残差网络中的跳跃连接

在残差网络（Residual Network）中，跳跃连接（Skip Connection）是一种特殊的连接方式，它将输入直接加到输出上，形成 `y = F(x) + x` 的结构。

**作用**：
- 解决深层网络的梯度消失问题
- 保留原始信号，避免信号逐层衰减
- 使得网络可以训练得更深

### 1.2 文档工程体系中的跳跃连接

在文档工程体系中，跳跃连接的思想被应用到文档层级关系中：

**传统方式**：
```
PRD → Sub-PRD → LLD → API → 数据库 → 测试
```

信号逐层传递，每层都可能衰减或失真。

**跳跃连接方式**：
```
PRD ←─────────────────────────────────────┐
  │                                        │
  ▼                                        │
Sub-PRD ←─────────────────────────────┐    │
  │                                    │    │
  ▼                                    │    │
LLD ←─────────────────────────────┐    │    │
  │                                │    │    │
  ▼                                │    │    │
API ←─────────────────────────┐    │    │    │
  │                            │    │    │    │
  ▼                            │    │    │    │
数据库 ←───────────────────┐    │    │    │    │
  │                        │    │    │    │    │
  ▼                        │    │    │    │    │
测试 ──────────────────────┴────┴────┴────┴────┘
```

每个下游文档都直接锚定上游原文，避免信号逐层衰减。

## 二、实现方式

### 2.1 需求溯源区

下游文档（LLD/API 合约/数据模型）必须包含"需求溯源区"，直接锚定上游原文。

**溯源区格式**：

```markdown
## 需求溯源区

| 原子需求 ID | 语义要素 | 来源（原文） | 推导链 |
|------------|---------|-------------|--------|
| @req-id: R001 | {要素描述} | {PRD/Sub-PRD 原文} | {设计决策推理过程} |
```

### 2.2 Grounding

溯源区内容必须逐字复制上游原文，不得改写。

**规则**：
- 禁止"大意相同"式的改写
- 必须原文复制
- 如需引用，保留原文引用格式

### 2.3 推导链

每个设计决策必须包含推导链（CoT），可追溯到原始需求。

**推导链格式**：

```markdown
### 推导链 1：@req-id: R001

**需求**：{PRD/Sub-PRD 原文}

**约束**：
- {约束 1}
- {约束 2}

**方案**：
- {方案描述}

**理由**：
- {为什么选择这个方案}
- {与其他方案的对比}

**结论**：
- {最终设计决策}
```

## 三、效果

### 3.1 信号保真

- 需求信号不会在文档层级间衰减
- 每个设计决策都可追溯到原始需求
- 审计时可以快速定位偏差来源

### 3.2 偏差可检测

- 覆盖率：检查需求要素是否在下游有承接
- 术语一致率：检查字段名和术语是否符合字典
- 数据流违例：检查模块依赖是否遵循架构

### 3.3 偏差可修正

- 偏差可以转化为结构化补丁
- 补丁可以被审计和签发
- 补丁可以被应用和回滚

---

## 四、使用场景

### 4.1 文档生成

- LLD 生成时，必须包含需求溯源区
- API 合约生成时，必须包含需求溯源区
- 数据模型生成时，必须包含需求溯源区

### 4.2 文档审计

- 残差审计时，检查溯源区是否存在
- 检查溯源区是否覆盖所有相关原子需求
- 检查溯源区内容是否逐字复制上游原文

### 4.3 文档修正

- 修改文档时，必须以结构化补丁形式提交
- 补丁必须包含 reason、source_requirement、impact
- 补丁可以被审计和签发
