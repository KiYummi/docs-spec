# 残差审计策略详解

> **适用范围**：文档工程体系（docs-spec）
> **最后更新**：2026-05-16
> **涉及思想**：残差网络、残差信号、跳跃连接、补丁式修正

---

## 一、概念

### 1.1 残差网络中的残差学习

在残差网络中，残差学习（Residual Learning）是一种特殊的网络结构，它学习的是"偏差"而非"绝对值"。

**核心公式**：
```
y = F(x) + x
```

其中：
- x：输入
- F(x)：偏差（残差）
- y：输出

**作用**：
- 学习偏差比学习绝对值更容易
- 可以检测和修正偏差
- 可以保留原始信号

### 1.2 文档工程体系中的残差审计

在文档工程体系中，残差审计的思想被应用到文档质量检查中：

**核心思路**：
- 检查"偏差"而非"绝对值"
- 检查需求是否在下游有承接
- 检查术语是否符合字典
- 检查数据流是否遵循架构

**作用**：
- 偏差可检测
- 偏差可追溯
- 偏差可修正

## 二、审计维度

### 2.1 语义覆盖率审计

**目标**：检查每条原子需求的语义要素是否在下游文档中有承接。

**检查项**：
- 原子需求基线是否提取完整
- 语义要素是否在下游有承接
- 覆盖率是否达标

**覆盖率要求**：
- P0 需求：100% 覆盖
- P1 需求：≥ 90% 覆盖
- P2 需求：≥ 70% 覆盖

### 2.2 术语一致性扫描

**目标**：检查字段名和术语是否符合字典定义。

**检查项**：
- 字段名是否在 metadata-dictionary 中
- 术语是否在 glossary 中
- 命名一致性是否达标
- 格式一致性是否达标

### 2.3 数据流一致性检查

**目标**：检查模块依赖是否遵循架构设计。

**检查项**：
- 模块依赖方向是否正确
- 数据流向是否一致
- 是否存在循环依赖
- 跨域调用是否使用事件模型

### 2.4 短路校验

**目标**：原始需求直通底层制品，验证完整性。

**检查项**：
- PRD → Sub-PRD 是否有承接
- Sub-PRD → LLD 是否有承接
- LLD → API 是否有承接
- API → 数据库是否有承接
- 数据库 → 测试是否有承接

---

## 三、审计流程

### 3.1 提取基线

1. 从 PRD 中提取原子需求清单
2. 每条需求分配唯一 ID
3. 语义要素必须完整

### 3.2 逐层比对

1. PRD → Sub-PRD
2. Sub-PRD → LLD
3. LLD → API
4. API → 数据库
5. 数据库 → 测试

### 3.3 短路校验

1. 原始需求直通底层制品
2. 验证完整性
3. 检查每一层是否有对应的需求 ID 标注

### 3.4 输出补丁

1. 将审计发现转化为结构化补丁
2. 补丁必须包含 reason、source_requirement、impact
3. 补丁可以被审计和签发

---

## 四、推理链要求（CoT）

### 4.1 推理链格式

```
发现：{问题描述}
证据：{文件路径}:{行号} → {具体内容}
推理：{推理过程}
结论：{明确结论}
```

### 4.2 推理链规则

- 每个审计结论必须包含推理过程
- 不接受"经验判断"式的结论
- 推理链必须可追溯

---

## 五、触发场景

| 场景 | 触发者 | 说明 |
|------|--------|------|
| 文档生成后 | coordinator | 检查新生成文档的保真度 |
| Gate 通过前 | coordinator | 强制门禁检查 |
| 重大修订后 | coordinator | 检查修订是否引入偏差 |
| 定期巡检 | coordinator | 全链路健康检查 |

---

## 六、效果

### 6.1 偏差可检测

- 覆盖率：检查需求要素是否在下游有承接
- 术语一致率：检查字段名和术语是否符合字典
- 数据流违例：检查模块依赖是否遵循架构

### 6.2 偏差可追溯

- 每个偏差都有明确来源
- 每个偏差都有证据
- 每个偏差都有推理链

### 6.3 偏差可修正

- 偏差可以转化为结构化补丁
- 补丁可以被审计和签发
- 补丁可以被应用和回滚
